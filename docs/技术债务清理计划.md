### 原则

1. **最小破坏原则**：保持对外API不变

3. **向后兼容**：不影响现有mod和玩家存档
4. **测试驱动**：每次拆分后验证功能

### 1.2 拆分方案

#### 方案 A：按职责完全拆分（推荐）

**新结构**：
```
RSS/Environment/
├── SCR_EnvironmentFactor.c (核心管理 + 对外API ~250行)
├── SCR_WeatherSystem.c (天气系统 ~300行)
├── SCR_TemperatureCalculator.c (温度计算 ~450行)
├── SCR_MudSystem.c (泥泞系统 ~200行)
└── SCR_EnvironmentIndoorDetection.c (已存在，保持)
```

**EnvironmentFactor 新角色**：
- 管理器/协调器
- 持有各子系统实例
- 提供统一对外 API
- 缓存管理

**示例代码**：
```cpp
class EnvironmentFactor
{
    // 子系统
    protected ref SCR_WeatherSystem m_pWeatherSystem;
    protected ref SCR_TemperatureCalculator m_pTempCalculator;
    protected ref SCR_MudSystem m_pMudSystem;
    protected ref SCR_EnvironmentIndoorDetection m_pIndoorDetector;
    
    // 对外API（保持不变）
    float GetHeatStressMultiplier() {
        return m_pTempCalculator.GetHeatStressMultiplier();
    }
    
    float GetWindDrag() {
        return m_pWeatherSystem.GetWindDrag();
    }
    
    // ... 其他API不变
}
```

**优点**：
- ✅ 职责清晰，单一职责原则
- ✅ 易于测试
- ✅ 未来扩展容易（如添加雷暴系统）

**缺点**：
- ⚠️ 需要创建 4 个新文件
- ⚠️ 需要仔细处理状态共享
