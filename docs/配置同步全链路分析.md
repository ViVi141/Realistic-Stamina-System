# RSS 配置同步全链路分析：客户端请求 → 服务器分发 → 客户端应用

本文档从**客户端读取/请求**、**服务器读取与分发**、**客户端应用**三阶段对配置同步做端到端梳理，便于排查 HUD/预设等不生效问题。

---

## 一、总览：数据流与角色

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│  服务器                                                                          │
│  ┌──────────────────┐     GetSettings()      ┌─────────────────────────────┐   │
│  │ $profile:         │ ───────────────────►  │ SCR_RSS_ConfigManager       │   │
│  │ RealisticStamina  │   Load() 读 JSON       │ (单例 m_Settings)            │   │
│  │ System.json       │                        │ 唯一配置源，供下发用          │   │
│  └──────────────────┘                        └──────────────┬──────────────┘   │
│                                                              │                  │
│                                         BuildPresetArray      │ BuildSettingsArrays
│                                         WriteSettingsToArrays │                  │
│                                                              ▼                  │
│  ┌──────────────────────────────────────────────────────────────────────────┐   │
│  │ RPC 下发：RPC_SendFullConfigOwner(单客户端) / RPC_SendFullConfigBroadcast  │   │
│  │ 参数：configVersion, selectedPreset, 4×预设数组, floatSettings,            │   │
│  │       intSettings, boolSettings（含 m_bHintDisplayEnabled）               │   │
│  └──────────────────────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────────────────────┘
                                        │
                                        │ 网络 RPC (Reliable)
                                        ▼
┌─────────────────────────────────────────────────────────────────────────────────┐
│  客户端                                                                          │
│  ┌─────────────────────────────┐     ApplyFullConfig() / RPC_ClientReceiveConfig  │
│  │ SCR_RSS_ConfigManager       │ ◄── 使用收到的数组写回单例 m_Settings           │
│  │ 不读本地 JSON，初值为       │     ApplyParamsFromArray + ApplySettingsFromArrays│
│  │ 内存默认（HUD 默认 false）  │                                                  │
│  └──────────────┬──────────────┘                                                  │
│                 │                                                                 │
│                 │ GetSettings() 供全模组使用（体力逻辑、HUD、调试等）              │
│                 ▼                                                                 │
│  ┌──────────────────┐   m_bHintDisplayEnabled   ┌─────────────────────────────┐   │
│  │ 体力/预设/倍率等  │ ───────────────────────► │ SCR_StaminaHUDComponent     │   │
│  │ 全部来自服务器   │                           │ Init() / Destroy()         │   │
│  └──────────────────┘                           └─────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────────────────────┘
```

- **服务器**：唯一从磁盘读 JSON 的一方，`GetSettings()` 返回的 `m_Settings` 是下发的唯一来源。
- **客户端**：不读本地 JSON，初始为内存默认；收到 RPC 后通过 `ApplyFullConfig` 或 `RPC_ClientReceiveConfig` 写回本机 `ConfigManager` 的 `m_Settings`，后续所有逻辑（含 HUD）都基于该单例。

## 二、阶段一：配置的“读取”与来源

### 2.1 服务器端“读取”（唯一持久化源）

| 步骤 | 位置 | 说明 |
|------|------|------|
| 触发 | `PlayerBase` EOnInit，`Replication.IsServer()` 为 true | 首个服务器端角色控制器初始化时执行一次 |
| 调用 | `SCR_RSS_ConfigManager.Load()` | 若 `m_Settings` 为空，`GetSettings()` 也会间接调用 `Load()` |
| 路径 | `CONFIG_PATH` = `$profile:RealisticStaminaSystem.json` | 服务器 profile 目录，非客户端目录 |
| 有文件 | `SCR_JsonLoadContext.LoadFromFile` + `ReadValue("", m_Settings)` | 反序列化到单例 `m_Settings`，含 `m_bHintDisplayEnabled` 等 |
| 无文件 | 创建默认 `m_Settings` 并 `Save()` 写回 | 默认 `m_bHintDisplayEnabled = false` |
| 工作台 | `#ifdef WORKBENCH` 不读文件，用嵌入预设，强制 HUD=true、Debug=true | 不依赖 JSON |

要点：**HUD 是否在服务器“开启”完全由该 JSON 中的 `m_bHintDisplayEnabled` 决定**；键名需与类成员一致（如 `"m_bHintDisplayEnabled": true`）。

### 2.2 客户端“读取”（仅内存默认，不读 JSON）

| 步骤 | 位置 | 说明 |
|------|------|------|
| 触发 | `SCR_RSS_ConfigManager.Load()` 在客户端被调用时（如首次 `GetSettings()`） | 客户端分支 `if (!Replication.IsServer())` |
| 行为 | 不执行 `LoadFromFile`，直接构造内存默认 | `m_Settings.m_bHintDisplayEnabled = false` 等 |
| 持久化 | 客户端不写 JSON（`CanWriteConfig()` 为 false） | `Save()` 在客户端只更新内存缓存并 return |

结论：**客户端没有任何“从本地 JSON 读取”的路径**，只存在“内存默认 → 等待服务器下发覆盖”。

## 三、阶段二：服务器分发（何时、以何形式下发）

### 3.1 下发方式概览

| 方式 | RPC | 触发时机 | 接收方 |
|------|-----|----------|--------|
| 客户端主动请求 | `RPC_ServerRequestConfig` → 服务器回复 `RPC_SendFullConfigOwner` | 客户端 1s 延迟 + 每 5s 周期 | 请求者（Owner） |
| 服务器配置变更 | `OnConfigChanged()` → `RPC_SendFullConfigBroadcast` | 服务器 Save() 且 DetectConfigChanges 为 true | 所有客户端 |
| 服务器角色就绪推送 | `PushConfigToOwnerClient` → `RPC_SendFullConfigOwner` | 服务器上任一角色 OnInit 后 2.5s（该角色归属谁则推给谁） | 本实体 Owner |
| 服务器定期广播 | `BroadcastConfigToAllClients` → `RPC_SendFullConfigBroadcast` | 开服 3s 后首次，之后每 10s | 所有客户端 |
| 精简键值 RPC | `RPC_SendConfigData`（仅部分字段） | 当前代码中未见调用方 | Broadcast |

**注意**：玩家连接到服务器时**并不代表已拥有角色**；角色在服务器上创建时才会触发本组件的 OnInit。因此配置下发不依赖“连接即发”，而是依赖：角色创建后的 Owner 推送、定期广播、以及客户端在**已有角色**后的周期请求。

### 3.2 路径 A：客户端请求 → 服务器单播完整配置

```
客户端                                                    服务器
   │                                                         │
   │  EOnInit 后 1s: CallLater(RequestServerConfig, 1000)    │
   │  RequestServerConfig()                                  │
   │  ──────── RPC_ServerRequestConfig() ──────────────────►  │
   │                                                         │ GetSettings()
   │                                                         │ BuildPresetArray ×4
   │                                                         │ BuildSettingsArrays
   │                                                         │ (含 boolSettings[3]=m_bHintDisplayEnabled)
   │  ◄──────── RPC_SendFullConfigOwner(...) ───────────────  │ RplRcver.Owner
   │                                                         │
   │  ApplyFullConfig(...)  [见阶段三]                        │
```

- 触发：`PlayerBase.EOnInit` 中 `CallLater(RequestServerConfig, 1000, false)`（仅客户端）。
- 周期：`UpdateServerConfigSync()` 每 `SERVER_CONFIG_SYNC_INTERVAL`（5s）再调 `RequestServerConfig()`，实现定期拉取。

### 3.3 路径 B：服务器配置变更 → 广播完整配置

```
服务器                                                     客户端
   │                                                         │
   │  Save() [仅服务器写 JSON 后]                              │
   │  DetectConfigChanges()                                  │
   │  NotifyConfigChanges()                                  │
   │  → 对每个注册的 listener 调用 controller.OnConfigChanged()
   │  OnConfigChanged() [服务器侧]                           │
   │  GetSettings() + Build*Arrays                           │
   │  RPC_SendFullConfigBroadcast(...)                       │
   │  ───────────────────────────────────────────────────►   │
   │                                                         │ ApplyFullConfig(...)
```

- 触发：服务器侧 `SCR_RSS_ConfigManager.Save()` 执行且 `DetectConfigChanges()` 为 true 时调用 `NotifyConfigChanges()`；监听器在首个服务器角色控制器 EOnInit 时通过 `RegisterConfigChangeListener(owner)` 注册。
- 注意：**客户端**调用 `Save()` 时 `CanWriteConfig()` 为 false，不会写盘也不会触发 `NotifyConfigChanges`，因此不会产生此广播。

### 3.4 服务器侧组包：完整配置内容

- **预设四组**：`BuildPresetArray(m_EliteStandard|StandardMilsim|TacticalAction|Custom)` → 各 45 个 float（`WriteParamsToArray`）。
- **通用设置**：`BuildSettingsArrays` → `SCR_RSS_Settings.WriteSettingsToArrays`：
  - **floatSettings**：17 个（如 m_fHintDuration, m_fStaminaDrainMultiplier, ...）。
  - **intSettings**：4 个（m_iDebugUpdateInterval, m_iHintUpdateInterval, m_iTerrainUpdateInterval, m_iEnvironmentUpdateInterval）。
  - **boolSettings**：14 个，**第 4 项（index 3）为 `m_bHintDisplayEnabled`**，顺序与 `ApplySettingsFromArrays` 的 `bools[bi++]` 一致。

服务器从 `GetSettings()` 拿到的即磁盘 JSON 反序列化结果（或默认值），**下发内容与服务器当前内存配置一致**。

## 四、阶段三：客户端应用（收到 RPC 后如何生效）

### 4.1 入口：两种 RPC 落地到同一套“应用”逻辑

| RPC | 客户端执行 | 说明 |
|-----|------------|------|
| `RPC_SendFullConfigOwner` | `ApplyFullConfig(configVersion, selectedPreset, eliteParams, standardParams, tacticalParams, customParams, floatSettings, intSettings, boolSettings)` | 单播完整配置 |
| `RPC_SendFullConfigBroadcast` | 同上 `ApplyFullConfig(...)` | 广播完整配置 |

二者参数一致，仅接收范围不同；下面以 `ApplyFullConfig` 为代表说明“应用”全流程。

### 4.2 ApplyFullConfig 步骤（仅客户端执行）

```c
// PlayerBase.c
protected void ApplyFullConfig(...)
{
    if (Replication.IsServer()) return;   // 服务器不应用

    SCR_RSS_Settings settings = SCR_RSS_ConfigManager.GetSettings();
    if (!settings) settings = new SCR_RSS_Settings();

    // 1) 预设参数写回
    SCR_RSS_Settings.ApplyParamsFromArray(settings.m_EliteStandard, eliteParams);
    // ... 其余三组预设同理

    // 2) 通用设置写回（含 m_bHintDisplayEnabled = bools[3]）
    SCR_RSS_Settings.ApplySettingsFromArrays(settings, floatSettings, intSettings, boolSettings);

    // 3) 版本与预设名
    settings.m_sConfigVersion = configVersion;
    settings.m_sSelectedPreset = selectedPreset;

    // 4) 仅更新缓存与“已应用服务器配置”标志，不写盘（ApplyServerConfigNoWrite），避免客户端/主机在收到下发时写 JSON）
    SCR_RSS_ConfigManager.ApplyServerConfigNoWrite();

    // 5) HUD 与配置一致
    if (settings.m_bHintDisplayEnabled)
        SCR_StaminaHUDComponent.Init();
    else
        SCR_StaminaHUDComponent.Destroy();
}
```

- **写回对象**：始终是客户端 `ConfigManager` 的单例 `m_Settings`（或新建后由 `GetSettings()` 后续返回的同一实例）。
- **不写盘**：应用下发配置时只调用 `ApplyServerConfigNoWrite()`（更新缓存 + 设“已应用服务器配置”），不调用 `Save()`，因此客户端或主机在“收到配置”的瞬间不会创建/更新硬盘上的 JSON，避免与服务器下发时间戳一致造成的误解。
- **bool 顺序**：`ApplySettingsFromArrays` 中 `m_bHintDisplayEnabled = bools[bi++]` 与服务器 `WriteSettingsToArrays` 的插入顺序一致（第 4 个 bool），因此服务器 JSON 中 HUD 开关会正确传到客户端并写回。
- **HUD**：应用后立即根据 `m_bHintDisplayEnabled` 调用 `SCR_StaminaHUDComponent.Init()` 或 `Destroy()`，保证界面与配置一致。

### 4.3 另一条应用路径：RPC_ClientReceiveConfig（精简键值）

- 用于只传部分字段（如 version、preset、debug/hint 开关、倍率、部分间隔）。
- 客户端若已应用过且非强制，会按版本号决定是否覆盖；应用后同样根据 `m_bHintDisplayEnabled` 调用 `Init()`/`Destroy()`。
- 当前代码中**未见该 RPC 的发送方**，仅作备用/扩展用；主路径仍是“完整配置”的 Owner/Broadcast。

### 4.4 应用后的使用处

- **体力/预设/倍率**：所有游戏逻辑通过 `SCR_RSS_ConfigManager.GetSettings()` 取当前预设与参数，均为“已应用后的客户端单例”。
- **HUD**：`SCR_StaminaHUDComponent.Init()` 内部再次 `GetSettings()`，若 `m_bHintDisplayEnabled == true` 才创建 HUD；否则直接 return。
- **延迟补显**：若配置在 `InitStaminaHUD`（1s）之后才到达，应用时会已调用 `Init()`；若仍未创建，约 3s 的 `SyncHUDToConfig()` 会再次根据当前 `m_bHintDisplayEnabled` 执行 `Init()`/`Destroy()`，并可在未应用时再次 `RequestServerConfig()`。

## 五、全链路时序简图（客户端首次进入）

```
 时间    客户端                              网络/服务器
 ─────────────────────────────────────────────────────────────────────
  0ms    EOnInit
        • 不读 JSON，GetSettings() 用内存默认（HUD=false）
 500ms   StartSystem
 1000ms  RequestServerConfig() ────────────► RPC_ServerRequestConfig
        InitStaminaHUD()                    服务器 GetSettings()
        若此时尚未收到配置，Init() 内          Build*Arrays
        m_bHintDisplayEnabled 仍 false，      RPC_SendFullConfigOwner ──►
        不创建 HUD
 ~1xxx   ◄────────────────────────────────── 客户端收到 RPC
        ApplyFullConfig(...)
        m_Settings 写回（含 HUD=true）
        SCR_StaminaHUDComponent.Init() → 创建 HUD
 3000ms  SyncHUDToConfig()
        若此前未应用则再 RequestServerConfig；
        若已应用则按当前设置补 Init/Destroy
 5000ms  UpdateServerConfigSync() → RequestServerConfig()（周期重复）
```

## 六、排查清单（服务器开 HUD 但客户端不显示）

按链路逐项核对：

1. **服务器“读取”**
   - 服务器 profile 下是否存在 `RealisticStaminaSystem.json`？
   - JSON 中是否为 `"m_bHintDisplayEnabled": true`（键名、布尔值正确）？
   - 若文件不存在，服务器会建默认并写回，默认 HUD 为 false。

2. **服务器“分发”**
   - 客户端是否在 1s 后发出请求？可看客户端日志是否有 `[RSS] Client requesting server config`。
   - 服务器是否回复？可看服务器日志 `[RSS] Sync config to client (owner request): ...`。
   - 若依赖“配置变更广播”，需确认服务器在本次会话中执行过 `Save()` 且发生了变更（如改预设/改 JSON 后重载并保存）。

3. **客户端“应用”**
   - 客户端是否打印 `[RSS] Applied full server config: ... HUD(hintDisplayEnabled)=true`？若为 false，说明收到的配置里 HUD 为关。
   - 若为 true 仍无 HUD：检查 3s 的 `SyncHUDToConfig` 是否执行、是否有 UI/布局或其它逻辑再次关闭 HUD。

4. **HandleClientConfigRequest**
   - 当前工程中**无调用方**；若未来通过其它入口“按需推送给某客户端”，可走该函数发 `RPC_SendFullConfigOwner`，逻辑与请求-回复路径一致。

## 七、相关文件与常量

| 文件 | 职责 |
|------|------|
| `scripts/Game/PlayerBase.c` | EOnInit 调 Load/RequestServerConfig；RPC 定义与 ApplyFullConfig；InitStaminaHUD、SyncHUDToConfig |
| `scripts/Game/Components/Stamina/SCR_RSS_ConfigManager.c` | Load/Save、CONFIG_PATH、NotifyConfigChanges、CanWriteConfig、监听器 |
| `scripts/Game/Components/Stamina/SCR_RSS_Settings.c` | WriteSettingsToArrays / ApplySettingsFromArrays、PARAMS_ARRAY_SIZE、SETTINGS_*_SIZE |
| `scripts/Game/Components/Stamina/SCR_StaminaHUDComponent.c` | Init/Destroy、根据 m_bHintDisplayEnabled 创建/销毁 HUD |

| 常量 | 值 | 含义 |
|------|-----|------|
| CONFIG_PATH | `$profile:RealisticStaminaSystem.json` | 服务器（及工作台）唯一读写的 JSON 路径 |
| SERVER_CONFIG_SYNC_INTERVAL | 5.0 秒 | 客户端周期请求服务器配置间隔 |
| HUD_SYNC_DELAY_MS | 3000 ms | 客户端延迟同步 HUD 与配置（补显） |

以上即从**客户端读取（实为请求）→ 服务器分发 → 客户端应用**的完整链路说明，可用于实现与排错时对照。
