# RSS 配置与体力同步全链路分析：服务器计算 → 客户端接收

本文档从**服务器权威计算**、**服务器分发**、**客户端接收显示**三阶段对 RSS 链路做端到端梳理。当前实现为**彻底服务器权威**：客户端不参与体力真值计算，不上报体力，仅负责控制角色与显示服务器下发状态。

---

## 一、总览：数据流与角色

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│  服务器                                                                          │
│  ┌──────────────────┐     GetSettings()      ┌─────────────────────────────┐   │
│  │ $profile:         │ ───────────────────►  │ SCR_RSS_ConfigManager       │   │
│  │ RealisticStamina  │   Load() 读 JSON       │ (单例 m_Settings)            │   │
│  │ System.json       │                        │ 唯一配置源，供下发用          │   │
│  └──────────────────┘                        └──────────────┬──────────────┘   │
│                                                              │                  │
│                                         BuildPresetArray      │ BuildSettingsArrays
│                                         WriteSettingsToArrays │                  │
│                                                              ▼                  │
│  ┌──────────────────────────────────────────────────────────────────────────┐   │
│  │ 主路径1：持久实体上的 SCR_RSS_ConfigSyncComponent（RplProp NoOwner）         │   │
│  │        → 下发服务器配置（HUD 开关、预设索引）                               │   │
│  │ 主路径2：角色组件 RplProp（OwnerOnly）                                      │   │
│  │        → 下发服务器权威体力与速度（m_fServerStaminaPercent / Speed）        │   │
│  └──────────────────────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────────────────────┘
                                        │
                        ┌───────────────┴───────────────┐
                        │ 复制属性同步 (RplProp)            │
                        │ 无体力上报 RPC 依赖                │
                        └───────────────┬───────────────┘
                                        ▼
┌─────────────────────────────────────────────────────────────────────────────────┐
│  客户端                                                                          │
│  ┌─────────────────────────────┐     OnServerConfigReplicated / OnServerSpeed  │
│  │ SCR_RSS_ConfigManager       │ ◄── Replicated 回调写 m_Settings；            │
│  │ 不读本地 JSON，初值为       │     OnServerStaminaReplicated 写体力组件        │
│  │ 内存默认（HUD 默认 false）  │                                                  │
│  └──────────────┬──────────────┘                                                  │
│                 │                                                                 │
│                 │ GetSettings() 供全模组使用（体力逻辑、HUD、调试等）              │
│                 ▼                                                                 │
│  ┌──────────────────┐   m_bHintDisplayEnabled   ┌─────────────────────────────┐   │
│  │ 体力/预设/倍率等  │ ───────────────────────► │ SCR_StaminaHUDComponent     │   │
│  │ 全部来自服务器   │                           │ Init() / Destroy()         │   │
│  └──────────────────┘                           └─────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────────────────────┘
```

- **服务器**：唯一从磁盘读 JSON 的一方，`GetSettings()` 返回的 `m_Settings` 是下发的唯一来源。配置由持久实体 `SCR_RSS_ConfigSyncComponent` 下发；体力与速度由角色组件 RplProp（`OwnerOnly`）下发，均由服务器计算并 `Replication.BumpMe()`。
- **客户端**：不读本地 JSON，初始为内存默认。客户端只控制角色输入，并在复制回调中接收与显示服务器下发状态；客户端不参与体力真值计算，也不上报体力。

## 二、阶段一：配置的“读取”与来源

### 2.1 服务器端“读取”（唯一持久化源）

| 步骤 | 位置 | 说明 |
|------|------|------|
| 触发 | `PlayerBase` EOnInit，`Replication.IsServer()` 为 true | 首个服务器端角色控制器初始化时执行一次 |
| 调用 | `SCR_RSS_ConfigManager.Load()` | 若 `m_Settings` 为空，`GetSettings()` 也会间接调用 `Load()` |
| 路径 | `CONFIG_PATH` = `$profile:RealisticStaminaSystem.json` | 服务器 profile 目录，非客户端目录 |
| 有文件 | `SCR_JsonLoadContext.LoadFromFile` + `ReadValue("", m_Settings)` | 反序列化到单例 `m_Settings`，含 `m_bHintDisplayEnabled` 等 |
| 无文件 | 创建默认 `m_Settings` 并 `Save()` 写回 | 默认 `m_bHintDisplayEnabled = false` |
| 工作台 | `#ifdef WORKBENCH` 不读文件，用嵌入预设，强制 HUD=true、Debug=true | 不依赖 JSON |

要点：**HUD 是否在服务器“开启”完全由该 JSON 中的 `m_bHintDisplayEnabled` 决定**；键名需与类成员一致（如 `"m_bHintDisplayEnabled": true`）。

### 2.2 客户端“读取”（仅内存默认，不读 JSON）

| 步骤 | 位置 | 说明 |
|------|------|------|
| 触发 | `SCR_RSS_ConfigManager.Load()` 在客户端被调用时（如首次 `GetSettings()`） | 客户端分支 `if (!Replication.IsServer())` |
| 行为 | 不执行 `LoadFromFile`，直接构造内存默认 | `m_Settings.m_bHintDisplayEnabled = false` 等 |
| 持久化 | 客户端不写 JSON（`CanWriteConfig()` 为 false） | `Save()` 在客户端只更新内存缓存并 return |

结论：**客户端没有任何“从本地 JSON 读取”的路径**，只存在“内存默认 → 等待服务器下发覆盖”。

## 三、阶段二：服务器分发（何时、以何形式下发）

### 3.1 下发方式概览：双主路径（配置 + 体力）均为服务器复制

**配置路径（持久实体）**：在场景中放置一个**带 RplComponent 的持久实体**（例如与 GameMode 同实体），并挂载 **SCR_RSS_ConfigSyncComponent**。服务器通过该组件 RplProp（`m_bServerHintEnabled`、`m_iServerPresetIndex`）向所有客户端复制配置。

**体力路径（角色实体）**：服务器在角色组件上计算并复制 `m_fServerStaminaPercent` 与 `m_fServerSpeedMultiplier`（`OwnerOnly`）。客户端仅在 `OnServerStaminaReplicated/OnServerSpeedReplicated` 中应用显示或本地体力组件，不做真值计算。

**部署要求**：场景中必须有带 `RplComponent + SCR_RSS_ConfigSyncComponent` 的持久实体；角色体力复制由角色组件内 RplProp 自动完成。

**RplProp 列表与 BumpMe 触发点**：

| RplProp | 类型 | 含义 | 服务器写入并 BumpMe 的时机 |
|---------|------|------|---------------------------|
| `m_bServerHintEnabled` | bool | HUD 开关 | `OnConfigChanged`、`PushConfigToOwnerClient`、体力回复时（与速度一同下发） |
| `m_iServerPresetIndex` | int | 0=EliteStandard, 1=StandardMilsim, 2=TacticalAction, 3=Custom | `OnConfigChanged`、`PushConfigToOwnerClient` |
| `m_fServerSpeedMultiplier` | float | 服务器校验后的速度倍数 | 体力回复路径（初次验证或偏差触发时） |

| 方式 | 主路径（RplProp） | 后备（RPC） | 接收方 |
|------|------------------|-------------|--------|
| 配置加载/变更 | `OnConfigChanged` → `SetServerConfigPropsAndBump()` | `RPC_SendFullConfigBroadcast` | Owner（RplProp）/ 所有客户端后过滤（RPC） |
| 角色就绪推送 | `PushConfigToOwnerClient` → `SetServerConfigPropsAndBump()` | `RPC_SendFullConfigBroadcast` | 同上 |
| 客户端请求 | （无单独 RplProp 推送；依赖上两项或体力路径） | `RPC_ServerRequestConfig` → `RPC_SendFullConfigBroadcast` | 同上 |
| 定期广播 | — | `BroadcastConfigToAllClients` 每 10s | 同上 |
| 体力路径 HUD/速度 | 设置 `m_fServerSpeedMultiplier`、`m_bServerHintEnabled` + `BumpMe()` | `RPC_ServerSyncSpeedMultiplier` | 同上 |

**预设与 JSON 结构**：当前**生效**的预设由 `m_sSelectedPreset` 指定（如 `StandardMilsim`=平衡）。JSON 文件**始终包含四个预设块**（`m_EliteStandard`、`m_StandardMilsim`、`m_TacticalAction`、`m_Custom`），并非“默认平衡则文件里只有平衡”；打开文件看到 `m_EliteStandard` 是正常现象，真正生效的是 `m_sSelectedPreset`。

**玩家与角色分离**：玩家是玩家，角色是角色；**只有复活/生成后玩家才操控角色**。连接时玩家可能尚未拥有角色，本组件挂在**角色**上，因此：
- 角色在服务器上创建（玩家复活等）时才会触发本组件 OnInit，此时才可能做“向该角色 Owner 单播”；
- 配置下发不依赖“连接即发”，而是依赖：**定期广播**（发往所有已连接客户端，与是否有角色无关）、角色创建后的 Owner 推送、以及客户端在**已有角色**后的周期请求。

### 3.2 路径 A：客户端请求 → 服务器单播完整配置

```
客户端                                                    服务器
   │                                                         │
   │  EOnInit 后 1s: CallLater(RequestServerConfig, 1000)    │
   │  RequestServerConfig()                                  │
   │  ──────── RPC_ServerRequestConfig() ──────────────────►  │
   │                                                         │ GetSettings()
   │                                                         │ BuildPresetArray ×4
   │                                                         │ BuildSettingsArrays
   │                                                         │ (含 boolSettings[3]=m_bHintDisplayEnabled)
   │  ◄──────── RPC_SendFullConfigBroadcast(...) ─────────────  │ RplRcver.Broadcast（仅本角色者应用）
   │                                                         │
   │  ApplyFullConfig(...)  [见阶段三]                        │
```

- 触发：`PlayerBase.EOnInit` 中 `CallLater(RequestServerConfig, 1000, false)`（仅客户端）。
- 周期：`UpdateServerConfigSync()` 每 `SERVER_CONFIG_SYNC_INTERVAL`（5s）再调 `RequestServerConfig()`，实现定期拉取。

### 3.3 路径 B：服务器配置变更 → 广播完整配置

```
服务器                                                     客户端
   │                                                         │
   │  Save() [仅服务器写 JSON 后]                              │
   │  DetectConfigChanges()                                  │
   │  NotifyConfigChanges()                                  │
   │  → 对每个注册的 listener 调用 controller.OnConfigChanged()
   │  OnConfigChanged() [服务器侧]                           │
   │  GetSettings() + Build*Arrays                           │
   │  RPC_SendFullConfigBroadcast(...)                       │
   │  ───────────────────────────────────────────────────►   │
   │                                                         │ ApplyFullConfig(...)
```

- 触发：服务器侧 `SCR_RSS_ConfigManager.Save()` 执行且 `DetectConfigChanges()` 为 true 时调用 `NotifyConfigChanges()`；监听器在首个服务器角色控制器 EOnInit 时通过 `RegisterConfigChangeListener(owner)` 注册。
- 注意：**客户端**调用 `Save()` 时 `CanWriteConfig()` 为 false，不会写盘也不会触发 `NotifyConfigChanges`，因此不会产生此广播。

### 3.4 服务器侧组包：完整配置内容

- **预设四组**：`BuildPresetArray(m_EliteStandard|StandardMilsim|TacticalAction|Custom)` → 各 45 个 float（`WriteParamsToArray`）。
- **通用设置**：`BuildSettingsArrays` → `SCR_RSS_Settings.WriteSettingsToArrays`：
  - **floatSettings**：17 个（如 m_fHintDuration, m_fStaminaDrainMultiplier, ...）。
  - **intSettings**：4 个（m_iDebugUpdateInterval, m_iHintUpdateInterval, m_iTerrainUpdateInterval, m_iEnvironmentUpdateInterval）。
  - **boolSettings**：14 个，**第 4 项（index 3）为 `m_bHintDisplayEnabled`**，顺序与 `ApplySettingsFromArrays` 的 `bools[bi++]` 一致。

服务器从 `GetSettings()` 拿到的即磁盘 JSON 反序列化结果（或默认值），**下发内容与服务器当前内存配置一致**。

## 四、阶段三：客户端应用（RplProp 回调与 RPC 后备）

### 4.1 主路径：RplProp 回调应用

| 回调 | 触发条件 | 客户端执行 |
|------|----------|------------|
| `OnServerConfigReplicated` | `m_bServerHintEnabled` 或 `m_iServerPresetIndex` 复制到本端 | 若 `!Replication.IsServer() && IsPlayerControlled()`：写 `GetSettings().m_bHintDisplayEnabled`、`m_sSelectedPreset`（由索引映射），`InitPresets(true)`、`ApplyServerConfigNoWrite()`，再 `SCR_StaminaHUDComponent.Init()` 或 `Destroy()` |
| `OnServerSpeedReplicated` | `m_fServerSpeedMultiplier` 复制到本端 | 若 `!Replication.IsServer() && IsPlayerControlled()`：`m_pNetworkSyncManager.SetServerValidatedSpeedMultiplier(m_fServerSpeedMultiplier)` |

预设索引与名称映射：0=EliteStandard, 1=StandardMilsim, 2=TacticalAction, 3=Custom（`GetPresetNameFromIndex` / `GetPresetIndexFromName` 在 PlayerBase 内）。

### 4.2 后备路径：RPC 落地到同一套“应用”逻辑

| RPC | 客户端执行 | 说明 |
|-----|------------|------|
| `RPC_SendFullConfigOwner` | 先判 `!Replication.IsServer() && IsPlayerControlled()` 再 `ApplyFullConfig(...)` | 保留兼容，实际下发已改用 Broadcast |
| `RPC_SendFullConfigBroadcast` | 同上，再 `ApplyFullConfig(...)` | 完整配置下发；仅本地操控角色应用 |
| `RPC_ServerSyncSpeedMultiplier` | 先判 `!Replication.IsServer() && IsPlayerControlled()` 再写速度与 HUD | 体力路径同步后备 |

所有“服务器→客户端”RPC 在客户端处理时都先做 `IsPlayerControlled()`，避免非本角色也应用配置。下面以 `ApplyFullConfig` 为代表说明“应用”全流程。

### 4.3 ApplyFullConfig 步骤（仅客户端执行，RPC 后备路径）

```c
// PlayerBase.c
protected void ApplyFullConfig(...)
{
    if (Replication.IsServer()) return;   // 服务器不应用

    SCR_RSS_Settings settings = SCR_RSS_ConfigManager.GetSettings();
    if (!settings) settings = new SCR_RSS_Settings();

    // 1) 预设参数写回
    SCR_RSS_Settings.ApplyParamsFromArray(settings.m_EliteStandard, eliteParams);
    // ... 其余三组预设同理

    // 2) 通用设置写回（含 m_bHintDisplayEnabled = bools[3]）
    SCR_RSS_Settings.ApplySettingsFromArrays(settings, floatSettings, intSettings, boolSettings);

    // 3) 版本与预设名
    settings.m_sConfigVersion = configVersion;
    settings.m_sSelectedPreset = selectedPreset;

    // 4) 仅更新缓存与“已应用服务器配置”标志，不写盘（ApplyServerConfigNoWrite），避免客户端/主机在收到下发时写 JSON）
    SCR_RSS_ConfigManager.ApplyServerConfigNoWrite();

    // 5) HUD 与配置一致
    if (settings.m_bHintDisplayEnabled)
        SCR_StaminaHUDComponent.Init();
    else
        SCR_StaminaHUDComponent.Destroy();
}
```

- **写回对象**：始终是客户端 `ConfigManager` 的单例 `m_Settings`（或新建后由 `GetSettings()` 后续返回的同一实例）。
- **不写盘**：应用下发配置时只调用 `ApplyServerConfigNoWrite()`（更新缓存 + 设“已应用服务器配置”），不调用 `Save()`，因此客户端或主机在“收到配置”的瞬间不会创建/更新硬盘上的 JSON，避免与服务器下发时间戳一致造成的误解。
- **bool 顺序**：`ApplySettingsFromArrays` 中 `m_bHintDisplayEnabled = bools[bi++]` 与服务器 `WriteSettingsToArrays` 的插入顺序一致（第 4 个 bool），因此服务器 JSON 中 HUD 开关会正确传到客户端并写回。
- **HUD**：应用后立即根据 `m_bHintDisplayEnabled` 调用 `SCR_StaminaHUDComponent.Init()` 或 `Destroy()`，保证界面与配置一致。

### 4.4 另一条应用路径：RPC_ClientReceiveConfig（精简键值）

- 用于只传部分字段（如 version、preset、debug/hint 开关、倍率、部分间隔）。
- 客户端若已应用过且非强制，会按版本号决定是否覆盖；应用后同样根据 `m_bHintDisplayEnabled` 调用 `Init()`/`Destroy()`。
- 当前代码中**未见该 RPC 的发送方**，仅作备用/扩展用；主路径仍是“完整配置”的 Owner/Broadcast。

### 4.5 应用后的使用处

- **体力/预设/倍率**：所有游戏逻辑通过 `SCR_RSS_ConfigManager.GetSettings()` 取当前预设与参数，均为“已应用后的客户端单例”。
- **HUD**：`SCR_StaminaHUDComponent.Init()` 内部再次 `GetSettings()`，若 `m_bHintDisplayEnabled == true` 才创建 HUD；否则直接 return。
- **延迟补显**：若配置在 `InitStaminaHUD`（1s）之后才到达，应用时会已调用 `Init()`；若仍未创建，约 3s 的 `SyncHUDToConfig()` 会再次根据当前 `m_bHintDisplayEnabled` 执行 `Init()`/`Destroy()`，并可在未应用时再次 `RequestServerConfig()`。

## 六、排查清单（服务器开 HUD 但客户端不显示）

按链路逐项核对：

1. **服务器“读取”**
   - 服务器 profile 下是否存在 `RealisticStaminaSystem.json`？
   - JSON 中是否为 `"m_bHintDisplayEnabled": true`（键名、布尔值正确）？
   - 若文件不存在，服务器会建默认并写回，默认 HUD 为 false。

2. **服务器“分发”**
   - **主路径（持久实体）**：场景中是否有带 RplComponent 的实体并挂载了 **SCR_RSS_ConfigSyncComponent**？若有，服务器在 `NotifyConfigChanges` 与 Bootstrap 的 Load 后会调用 `PushServerConfigToReplication()`；客户端收到复制会打印 `[RSS_ConfigSync] Applied from RplProp: preset=..., HUD=...`（调试开启时）。
   - **后备角色 RplProp**：角色组件上 `m_bServerHintEnabled`/`m_iServerPresetIndex` 等会由 `OnConfigChanged`、`PushConfigToOwnerClient` 设置并 BumpMe；客户端收到会打印 `[RSS] Applied config from RplProp: ...`。
   - **后备 RPC**：客户端是否在 1s 后发出请求？可看 `[RSS] Client requesting server config`；服务器是否回复可看 `[RSS] Sync config to client (owner request): ...`。

3. **客户端“应用”**
   - 先看是否出现 `[RSS] Applied config from RplProp`（主路径）；或 `[RSS] Applied full server config: ... HUD(hintDisplayEnabled)=true`（RPC 后备）。
   - 若为 true 仍无 HUD：检查 3s 的 `SyncHUDToConfig` 是否执行、是否有 UI/布局或其它逻辑再次关闭 HUD。

4. **HandleClientConfigRequest**
   - 当前工程中**无调用方**；若未来通过其它入口“按需推送给某客户端”，可走该函数发 `RPC_SendFullConfigBroadcast`，逻辑与请求-回复路径一致。

### 六.1 特殊场景：未控制角色与死亡复活

- **玩家尚未控制角色（大厅、观察、未重生）**
  - **主路径（持久实体）**：配置通过 `RplCondition.NoOwner` 复制到**所有客户端**，与是否拥有角色无关。客户端在 `SCR_RSS_ConfigSyncComponent.OnServerConfigReplicated` 中写回 ConfigManager 并 Init/Destroy HUD，不依赖角色。因此**无 BUG**。
  - **后备路径**：角色 RplProp 仅复制给该角色的 Owner，无角色时无法走该路径；RPC 需客户端主动请求（1s 后 RequestServerConfig）。若已部署持久实体同步组件，可不依赖后备。

- **死亡后复活到另一角色**
  - **配置（预设、HUD）**：ConfigManager 为单例，主路径在进服时已写入；复活后新角色不会覆盖或清空该状态。新角色在服务器端会注册为配置监听器、并在约 2.5s 收到 `PushConfigToOwnerClient`，客户端约 1s 会 `RequestServerConfig`，均为冗余同步，**无 BUG**。
  - **速度倍数**：`m_fServerSpeedMultiplier` 仅在服务器校验体力/速度时下发（RplProp + RPC）。新角色在首次校验前使用默认 1.0，校验后即与服务器一致，属预期行为。
  - **监听器泄漏（已修复）**：旧角色销毁时若未从 ConfigManager 注销，会导致监听器列表残留。本模组已在角色控制器 **EOnDelete** 中调用 `UnregisterConfigChangeListener(owner)`，避免死亡/销毁后仍被 NotifyConfigChanges 遍历到无效实体。

## 七、相关文件与常量

| `scripts/Game/SCR_RSS_ServerBootstrap.c` | OnGameStart 延迟调用 Load；Load 后若已注册同步组件则 PushServerConfigToReplication |
| `scripts/Game/PlayerBase.c` | EOnInit 调 Load/RequestServerConfig；RPC 定义与 ApplyFullConfig；InitStaminaHUD、SyncHUDToConfig |
| `scripts/Game/Components/Stamina/SCR_RSS_ConfigManager.c` | Load/Save、CONFIG_PATH、NotifyConfigChanges、GetConfigSyncComponent、Register/Unregister 同步组件 |
| `scripts/Game/Components/Stamina/SCR_RSS_ConfigSyncComponent.c` | 持久实体上的 RplProp 同步（m_bServerHintEnabled、m_iServerPresetIndex）；OnServerConfigReplicated 写回 ConfigManager 与 HUD |
| `scripts/Game/Components/Stamina/SCR_RSS_Settings.c` | WriteSettingsToArrays / ApplySettingsFromArrays、PARAMS_ARRAY_SIZE、SETTINGS_*_SIZE |
| `scripts/Game/Components/Stamina/SCR_StaminaHUDComponent.c` | Init/Destroy、根据 m_bHintDisplayEnabled 创建/销毁 HUD |

| 常量 | 值 | 含义 |
|------|-----|------|
| CONFIG_PATH | `$profile:RealisticStaminaSystem.json` | 服务器（及工作台）唯一读写的 JSON 路径 |
| SERVER_CONFIG_SYNC_INTERVAL | 5.0 秒 | 客户端周期请求服务器配置间隔 |
| HUD_SYNC_DELAY_MS | 3000 ms | 客户端延迟同步 HUD 与配置（补显） |

以上即从**客户端读取（实为请求）→ 服务器分发 → 客户端应用**的完整链路说明，可用于实现与排错时对照。
