# Python 优化器输出参数在测试脚本与游戏中表现差异 — 根因分析

## 问题描述

在使用**同一份**优化器输出的 JSON 参数时，`t1_benchmark.py`（或其它基于 `rss_digital_twin_fix.py` 的测试脚本）的仿真结果与**实际游戏**中的体力/恢复表现存在明显差距。

## 根因概览

| 类别 | 说明 |
|------|------|
| **1. 参数未进入游戏** | 优化器输出的部分参数未通过 `embed_json_to_c.py` 写入 C 预设，游戏端仍使用 C 内硬编码或旧预设值。 |
| **2. C 端硬编码与 JSON 不同步** | 部分在 Python/JSON 中可调、且被优化器改动的量在 C 中为 `static const`，游戏永远用固定值。 |
| **3. 嵌入与预设未执行/未对应** | 未在优化后执行嵌入脚本，或游戏使用的预设与测试所载 JSON 不一致。 |

---

## 根因 1：优化器输出的参数未写入 C（embed 映射缺失）

### 现象

- `tools/embed_json_to_c.py` 的 `param_mapping` 只包含**部分** JSON 键。
- 优化器输出的 JSON 中若包含映射里**没有**的键，这些值**不会**被写入 `SCR_RSS_Settings.c` 的任一预设。
- 测试脚本从 JSON 加载并传给数字孪生，因此**测试用的是优化后的值**；游戏从 C 预设/常量读，用的仍是**旧值或硬编码**，导致“同一参数”在两端不一致。

### 当前 JSON 有、但 embed 未映射的键（会直接导致差异）

| JSON 键 | 说明 | 游戏端现状 |
|--------|------|------------|
| `load_metabolic_dampening` | 负重代谢阻尼（T1 只承受负重额外代谢的该比例） | **C 端硬编码** `StaminaConstants.LOAD_METABOLIC_DAMPENING = 0.70`，不读配置。优化器常输出约 0.78 → 测试端消耗更少，游戏端消耗更多。 |
| `max_recovery_per_tick` | 每 tick 恢复率上限 | **C 端硬编码** `StaminaConstants.MAX_RECOVERY_PER_TICK = 0.0004`。若 JSON 中更大，测试端恢复更快，游戏端仍被 0.0004 限制。 |

只要优化器改动了上述两个量，就会在**消耗**和**恢复**上同时造成测试与游戏不一致。

---

## 根因 2：C 端硬编码导致“同一参数”无法在游戏中生效

### load_metabolic_dampening

- **Python 数字孪生**：`getattr(self.constants, 'LOAD_METABOLIC_DAMPENING', 0.70)`，从 JSON 加载后使用优化值（如 0.78）。
- **C 端**：`SCR_StaminaConstants.c` 中 `static const float LOAD_METABOLIC_DAMPENING = 0.70`；`SCR_StaminaUpdateCoordinator.c` 仅使用该常量。
- **结果**：游戏内负重带来的额外消耗始终按 0.70 阻尼计算，测试脚本按 0.78 → **游戏消耗大于测试**，同一场景下游戏体力掉得更快。

### max_recovery_per_tick

- **Python**：`getattr(c, 'MAX_RECOVERY_PER_TICK', 0.0004)` 并 `min(recovery_rate, max_cap)`。
- **C 端**：`SCR_StaminaConstants.c` 中 `static const float MAX_RECOVERY_PER_TICK = 0.0004`；`SCR_StaminaRecovery.c` 用该常量做恢复率上限。
- **结果**：若 JSON 中该值 > 0.0004，测试端恢复更快，游戏端仍被 0.0004 封顶 → **游戏恢复慢于测试**。

---

## 根因 3：嵌入未执行或预设不一致

### 嵌入未执行

- 优化器只生成/更新 `tools/optimized_rss_config_*_super.json`。
- 若未随后运行 `embed_json_to_c.py`，`SCR_RSS_Settings.c` 中的预设（如 `m_EliteStandard`）不会更新。
- 测试脚本直接读 JSON，所以测试用的是**新参数**；游戏读的是 C 里**上一次嵌入的旧值**，表现自然不同。

### 预设与 JSON 不对应

- `embed_json_to_c.py` 将 `optimized_rss_config_realism_super.json` → `EliteStandard`，`balanced` → `StandardMilsim`，`playability` → `TacticalAction`。
- 若服务器/客户端当前使用的是 **StandardMilsim** 或 **TacticalAction**，而测试/文档默认看的是 **realism/EliteStandard**，则“同一份 JSON”并未应用到当前游戏预设，也会表现为差异。

---

## 根因 4：Python 数字孪生中错误的 0.3 乘数（已修复）

- **位置**：`rss_digital_twin_fix.py` 中 `_compute_drain_and_recovery()` 最终计算 `total_drain = base * enc_mult * 0.3`。
- **C 端**：`SCR_StaminaConsumption.c` 中 `totalDrainRate = baseDrainRate * encumbranceStaminaDrainMultiplier`，**没有** 0.3。
- **结果**：测试脚本中的体力消耗只有游戏内的约 **30%**，表现为「 benchmark 体力掉得慢、游戏里掉得快」。
- **修复**：已去掉 Python 中错误的 `* 0.3`，使 `total_drain = base * enc_mult` 与 C 一致。

---

## 其它可能差异点（已核对或影响较小）

- **恢复阶段时长**：C 与 Python 均使用 `FAST_RECOVERY_DURATION_MINUTES = 0.4`、`MEDIUM_RECOVERY_START_MINUTES + MEDIUM_RECOVERY_DURATION_MINUTES = 5.4`，逻辑一致。
- **负重速度惩罚曲线**：C（`SCR_EncumbranceCache.c`）与 Python（`_encumbrance_speed_penalty_base`）使用相同的分段公式和 `(coeff/0.20)` 缩放，一致。
- **Pandolf 与静态站立**：公式与权重语义（总重 vs 负载）在调用链上已对齐；C 传入 `currentWeightWithWet`（总重），Python 使用 `current_weight`（总重），一致。

---

## 建议修复与流程

### 1. 让游戏端也使用优化器输出的 load_metabolic_dampening 和 max_recovery_per_tick（治本）— **已实现**

- 已在 `SCR_RSS_Params` 中增加字段：`load_metabolic_dampening`、`max_recovery_per_tick`。
- 已在 `SCR_StaminaConstants.c` 中增加 `GetLoadMetabolicDampening()`、`GetMaxRecoveryPerTick()`，消耗/恢复逻辑改为通过 Getter 读取预设。
- 已在 `embed_json_to_c.py` 的 `param_mapping` 中增加上述两键。
- **后续**：优化后执行 `python tools/embed_json_to_c.py` 并重新编译 addon，游戏即可与测试使用同一套数值。

### 2. 确保“同一参数”真的进游戏（流程）

- 每次用优化器生成/更新 JSON 后，执行：  
  `python tools/embed_json_to_c.py`  
  并重新编译 addon。
- 确认服务器/客户端使用的 RSS 预设与所测 JSON 对应（realism → EliteStandard 等）。

### 3. 若短期不改 C，保持测试与游戏一致（权宜）

- 在优化或评估时，将 JSON 中的 `load_metabolic_dampening` 固定为 `0.70`、`max_recovery_per_tick` 固定为 `0.0004`，与 C 硬编码一致；或不在优化目标中改动这两项，这样测试脚本与游戏在这两点上一致。

---

## 小结

- **主要差距**来自：优化器输出的 **load_metabolic_dampening** 和 **max_recovery_per_tick** 在测试脚本中生效，在游戏中因**未嵌入 + C 硬编码**而未生效。
- **次要差距**来自：优化后未运行嵌入脚本，或游戏使用的预设与测试所载 JSON 不对应。
- 若要“同一参数下测试与游戏表现一致”，需要让这两个量从 JSON → C 预设 → 游戏运行时全程打通，并保证嵌入流程与预设选择正确。

## 为何战斗周期图仍出现「最低体力 0%」

优化器在 `rss_super_pipeline.py` 中对**战术战斗周期**场景有硬约束剪枝：`test_type == '战术战斗周期'` 时，若存在任一体力点 \< 23%，会 `TrialPruned`，该组参数不会进入帕累托解。

当前仓库中的 `optimized_rss_config_realism_super.json`（以及绘图脚本 `plot_combat_cycle_30kg_en.py` 使用的配置）很可能是**在修复数字孪生 `total_drain = base * enc_mult * 0.3` 之前**跑出的优化结果。当时：

- 孪生里消耗被错误地乘了 0.3，**实际消耗约为现在的约 1/3**；
- 优化器在该「偏轻」消耗下评估战斗周期，很多参数能通过「最低体力 ≥ 23%」的约束；
- 导出并保存的预设，是按**旧、错误孪生**满足约束的。

修复 `* 0.3` 后，同一份 JSON 用**当前孪生**再跑同一战斗周期，消耗约变为约 3.3 倍，因此会出现在**接近阶段**（Approach）就掉到 0% 的情况，与图中「Min: 0.00%」一致。

**结论与建议：**

- 剪枝逻辑本身没问题，**最低体力 0% 来自「用旧预设 + 新孪生」** 的不一致。
- 若要在当前（已修 0.3）孪生下仍满足「战斗周期最低体力 ≥ 23%」，需要**用当前代码重新跑优化**，重新导出预设（并重新执行 `embed_json_to_c.py`）；旧 JSON 不应再期望在修后的孪生下满足该约束。
