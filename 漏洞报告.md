# 体力系统数值设计漏洞审查报告

基于最新版源码与文档，我从**公式逻辑**、**阈值设计**、**数值范围**、**配置协调**四个维度进行了全面检视。
以下条目包括代码位置、风险说明与建议修正方向。

---

## 🔴 高危问题（建议优先修复）

### 1. 恢复率计算逻辑不一致（3.7节）
**代码位置**：`SCR_RealisticStaminaSystem.c` `CalculateMultiDimensionalRecoveryRate()` 567–607

```c
float totalRecoveryRate = baseRecoveryRate * fitnessRecoveryMultiplier * … * marginalDecayMultiplier;
totalRecoveryRate = totalRecoveryRate - loadRecoveryPenalty;    // ← 乘法链后减去惩罚
totalRecoveryRate = Math.Max(totalRecoveryRate, 0.0);
```

- **问题**：乘法因子链之后突然出现减法项。若负重惩罚 > 乘积结果，会落到0；虽然`Math.Max`兜底，但逻辑不一致且调试困难。
- **风险**：优化器/修改时容易忽略“负恢复”情况；倒逼每次改参数都要分析减法前后上下界。
- **建议**：
  - 将负重惩罚改为乘法因子 `totalRecoveryRate *= (1.0 - loadRecoveryPenalty)`；
  - 或在减法前就设定 `lowerBound = baseRecoveryRate * … * …` 并确保负重惩罚被钳位。

### 2. 坡度处理逻辑分散且可能冲突

| 模块 | 功能 | 位置 |
|------|------|------|
| Pandolf坡度项 | 消耗计算 | `SCR_RealisticStaminaSystem.c` 882–948 |
| Santee下坡修正 | 消耗修正 | 同上 939–948 |
| Tobler徒步函数 | 速度调整 | `CalculateSlopeAdjustedTargetSpeed` 1000–1060 |
| CalculateGradeMultiplier | 未调用（遗留） | `SCR_RealisticStaminaSystem.c` 815–910 |

- **问题**：消耗侧和速度侧分别处理坡度，逻辑联系缺失。文档提示 `CalculateGradeMultiplier` 未调用，但未说明弃用理由。
- **风险**：开发或优化时易重复应用坡度惩罚或误以为某函数生效。
- **建议**：
  - 绘制雅克逊数据流图，明确 Pandolf+Santee 与 Tobler 的输入/输出边界。
  - 删除未使用的 `CalculateGradeMultiplier` 或在文档中注明其存档性质。

### 3. 配置值与硬编码值关系模糊

- **动态配置**：`SCR_RSS_Settings.c` 默认 `standing_recovery_multiplier = 2.0`。
- **工具/常量**：`tools/stamina_constants.py` 中设 `STANDING_RECOVERY_MULTIPLIER = 1.3`。
- **加载逻辑**：`SCR_RSS_ConfigManager` 在非 Custom 预设时会重写 JSON 为最新 optuna 值，而 Custom 时保留用户数据。

> 文档未说明“谁的值优先”，导致测试过程中预设切换时出现行为起伏。

- **建议**：在配置管理器注释与用户文档中明确写明“动态配置 > 工具硬编码；Custom 模式仅增量补全，永不覆盖”。

## 🟡 中危问题（影响体验/平衡）

### 4. Sprint启用阈值与意志力平台期过于接近
- 常量：`SPRINT_ENABLE_THRESHOLD = 0.20`；`WILLPOWER_THRESHOLD = 0.25`。
- 区间仅5%体力，玩家易出现“刚允许冲刺→冲刺耗光→速度骤降→无法再冲刺”的循环。
- **建议**：将二者间隔拓宽至10-15%，例如将 Sprint 阈值下调到0.15或将意志力起点上调至0.30。

### 5. 负重速度惩罚可能偏弱
- 30 kg负重时乘法项约5%，sprint情形7%——与“真实负重感”目标不符。
- **建议**：通过玩家测试验证，必要时将系数提高到0.3~0.4，或增加速度×负重交互惩罚。

### 6. 温度补偿公式生理合理性存疑
- 低温 `extra = 0.15 × dt²`；高温 `extra = 2.0 × dt`。
- 较大 dt 时低温侧额外功率反超高温；与运动生理学（冷应激常高于热应激）不一致。
- **建议**：参考 ACSM 等文献，调整系数比例，或将高温段改为平方增长。

## 🟢 低危问题（优化建议）

### 7. EPOC延迟时间过短
- 常量仅0.5 s，现实为分钟级。
- **建议**：若追求拟真，可延长3-5秒并让 `epocDrainRate` 随时间指数衰减。

### 8. 湿重积累/衰减速率比例
- 游泳 60s 达10 kg，衰减 30s 归零。
- 持续雨中或闪现场景可能出现“满→空→满”的抖动。
- **建议**：衰减速率与降雨强度联动，或在 `SwimmingStateManager.CalculateTotalWetWeight` 添加饱和逻辑。

### 9. 游泳模型P_survival项缺失公式
- 常量存在于表格 (`SWIMMING_SURVIVAL_STRESS_POWER = 15W`)，但文档未给出 P_total 表达式。
- **建议**：在文档相应段落补充 `P_total = P_static + P_dynamic + SWIMMING_SURVIVAL_STRESS_POWER`。

### 10. 镜头惯性过冲的非线性敏感
- `actualOvershoot = CAM_INERTIA_DECEL_OVERSHOOT_FORWARD_M × pow(loadFactor, CAM_INERTIA_OVERSHOOT_EXPONENT)`，指数1.5。
- 轻装与重装过冲差距达26倍，可能致手感突兀。
- **建议**：测试指数 1.0–1.2、或加入最小阈值以平滑过渡。
